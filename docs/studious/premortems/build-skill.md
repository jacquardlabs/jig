| # | Lane | Failure mode | Detection hint |

| 1 | technical | The Foreman's dispatch prompt leaks more than the task block + Read-first pointers + the discipline-skill trigger line — e.g. the full PLAN.md, the design doc, or a prior task's history bleeds in from the surrounding session — silently breaking the fresh-executor isolation the whole architecture (and the acceptance criteria) depend on. | Inspect the dispatch-prompt construction in `skills/build/SKILL.md` and a captured real dispatch; confirm it contains only the single task's checkpoint block verbatim, its named `Read first` pointers (as paths, not inlined content), and the one `task-execution-discipline` trigger line — nothing else from the calling session. |
| 2 | technical | The `status-flip` PASS-path integrity check is bypassed: the build lets the script accept a Foreman-supplied status on the `--results` path, or the Foreman routes a PASS through the `--status REPLAN\|ESCALATE` path, so a FAIL can be recorded as PASS. This silently voids "nothing signs off on itself" and the mis-transcription guard that justified adding the script (Alternatives considered #5). | Confirm `status-flip`'s PASS path derives the token solely from `--results` (`PASS` iff `results.json`'s `overall == "PASS"`) and never accepts a caller-supplied status alongside `--results`; grep the SKILL.md to confirm the Foreman never invokes `status-flip --status PASS`. |
| 3 | technical | The REPLAN idempotency carve-out is miswired — `status-flip` treats a pre-existing `REPLAN` suffix as terminal (like `PASS`/`ESCALATE`) and refuses on it. The human's ordinary resume path then dead-ends: the revised task's eventual PASS hits the stale `REPLAN` suffix, `status-flip` exits 2, the Foreman treats it as a usage error, retries once, refuses again, and PAUSEs with a misleading "usage error persisted" for a task that in fact passed. | Exercise a REPLAN → revise-block-by-hand → re-invoke `/build` → PASS sequence; confirm `status-flip` overwrites the `REPLAN` suffix in place and the task closes as PASS, rather than a spurious usage-error PAUSE. `PASS` and `ESCALATE` must still flip exactly once (refuse on re-flip); only `REPLAN` is overwritable. |
| 4 | technical | The Foreman runs `verify` before the executor's commit, or with the wrong `--since` SHA, making `evidence-capture`'s freshness check vacuous (`build-scripts` risk #1: a `now >= last-commit` check is meaningless while the task's work is still uncommitted). Evidence would then attest to nothing. | Confirm the SKILL.md sequence is strictly executor-commits → Foreman runs `verify --since <the executor's reported commit SHA>` → `evidence-capture`, never `verify` before the commit; check that the `--since` value is the SHA the executor returned, not the branch tip at dispatch time. |
| 5 | technical | The target project's `CLAUDE.md` names no baseline verification command and the Foreman proceeds anyway — guessing a test runner or silently skipping the baseline — shipping an unverified build. This is the exact silent, unverified-build failure "Standalone-capable" (graceful, never silent) rules out, and it strands `PRODUCT.md`'s secondary persona (no other jacquardlabs tooling installed). | Run `/build` against a project whose `CLAUDE.md` has no "Tests" (or equivalent) section; confirm it PAUSEs *before* any worktree is created, names the missing convention and the resume action (add a baseline command, re-invoke), and does not hardcode/guess a runner or add a second input flag. |
| 6 | product | A `PAUSED` verdict is reported bare (or names the wrong cause/action). `PAUSED` collapses four distinct triggers (missing/dirty baseline, REPLAN, a risk-tagged pre-dispatch ack, a persisted `verify`/`status-flip` usage error); a bare token leaves the human guessing which happened and what resumes `/build`, defeating the "Recommend one action; the human decides" posture. | Exercise each of the four PAUSE triggers; confirm every `PAUSED` report names, in the same message, which cause fired and the specific resume action for it (fix baseline and re-invoke; revise the block by hand and re-invoke; acknowledge the risk tag; fix the transcription bug and re-invoke). |
| 7 | product | The Foreman's judgment-based task-splitting still absorbs a coarser trailing heading (e.g. `## Not-here follow-ups`) into the last task's block — reproducing M0 friction finding 6 — so an executor receives non-task material in its dispatch, corrupting both its scope and the isolation claim in item 1. | Build a `PLAN.md` whose final `### Task` block is followed by a coarser `## ...` section; confirm that trailing section is excluded from every task's dispatched block and no executor's prompt contains it. |
