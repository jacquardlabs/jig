| 1 | technical | The new `skills/task-execution-discipline/` directory is a sixth top-level skill dir, but `tests/test_scaffold.py`'s `test_no_extra_top_level_skill_dirs` asserts `set(os.listdir(skills)) == set(EXPECTED_SKILLS)` where `EXPECTED_SKILLS = ("design","plan","build","finish","coach")` (line 45, assert line 97). The build fails that test, and the tempting fix — just appending the new dir to `EXPECTED_SKILLS` — blurs "user-invoked skill dirs" and "all skill dirs" into one set, which is what the design's Open question explicitly warns against. | At build/audit, run `python -m pytest tests/test_scaffold.py`. Confirm the fix splits the concept (a separate user-invoked vs. all-skills assertion) rather than widening `EXPECTED_SKILLS` to a 6-tuple. Grep the diff of `tests/test_scaffold.py` for a category distinction, not a one-token append. |
| 2 | technical | Model-invocation is carried *entirely* by the frontmatter `description` phrasing (there is no separate model-invoked field). If the drafted description reads as a command ("Run the discipline canon") rather than a trigger ("Use when about to implement a checkpoint item / about to claim Done-means"), Claude Code's loader never auto-routes to it and the skill silently never fires — the core purpose (fresh executors loading shared discipline) fails invisibly, with the file still present and structurally valid. | At acceptance, confirm the `description` uses "Use when…" trigger phrasing matching the model-invoked skills in this install (Studious/Superpowers `skills/*/SKILL.md`), and that the skill is actually *discoverable* (loads/lists), not merely that the frontmatter parses (design Open question #2). |
| 3 | technical | The canon claims to adapt Superpowers 6.1.1's `test-driven-development` and `verification-before-completion` ("~80% kept" per issue #6). Retargeting vocabulary can silently soften the source's two Iron Laws ("no production code without a failing test first"; "no completion claims without fresh verification evidence") into weaker jig-flavored prose, shipping discipline strictly weaker than the source it advertises preserving. | At audit/acceptance, diff the three shipped pillars against Superpowers' two source skills. Confirm both Iron Laws and the Red-Green-Refactor cycle survive the vocabulary retarget verbatim in force, not just in topic. |
| 4 | technical | The pillars hard-reference jig vocabulary whose source of truth is today the handoff, not shipped `SKILL.md` files (`cap`/`hold`, `test-backed` tier, `Do`/`Not here`/`Done means`/`Evidence`, `PASS`/`FIX`/`REPLAN`/`ESCALATE`). DESIGN.md flags M2–M6 haven't landed. If a later milestone renames any token, the skill's prose silently references stale vocabulary with no compiler to catch it. | At each later milestone that touches the checkpoint block or status enum, grep `skills/task-execution-discipline/SKILL.md` for the pre-rename token names; a hit is stale canon. |
| 5 | product | The description triggers on broad phrasing ("about to implement," "about to claim done"). With jig installed, a fresh executor in *any* repo — one with no checkpoint blocks at all — could auto-load this jig-specific discipline mid-task and inject `cap`/`Done-means`/status-flip vocabulary into a context that has none, confusing rather than helping. | At acceptance, confirm the trigger is gated on jig-build context (names checkpoint item / Done-means / the fresh-executor loop), not bare "writing implementation code," so it stays inert outside a jig `/build` session. |
| 6 | product | TDD-per-capability is scoped to `cap` items carrying the `test-backed` verification tier — a `cap` on the `script`/`probe` tier, or any `hold` item, is deliberately outside TDD's reach (DESIGN.md: only `test-backed` implies a test). If the drafted prose over-generalizes to "write a test for everything," it contradicts jig's own verification-tier model and drives executors to write tests the plan never asked for. | At acceptance, confirm the TDD pillar's applicability clause names the `test-backed` tier (and `cap` unit) explicitly as its precondition, not "every task" or "every capability." |
