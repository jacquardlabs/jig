# Pre-mortem — Build scripts (worktree setup, baseline check, verification re-run, evidence capture)

| # | Lane | Failure mode | Detection hint |
|---|------|--------------|----------------|
| 1 | technical | `evidence-capture`'s freshness check passes vacuously: it compares its timestamp against the branch's last *code commit*, but in the quick path the executor's work may still be uncommitted at capture time, so "last commit" is the baseline commit and any wall-clock-now stamp trivially satisfies `>=` — stale or copied-forward evidence from a prior attempt survives the gate the check exists to close. | At merge, read `scripts/evidence-capture`: confirm it errors when the task's files have uncommitted working-tree changes, or compares copied-artifact mtimes against the code commit rather than only stamping `now >= last-commit-ct`. |
| 2 | technical | Dirty-baseline stop leaves the worktree in place (by design), so re-running `/build` with the same branch name hits `worktree-setup`'s "fail loud on collision" path and blocks the retry until the user manually `git worktree remove`s — the two safety choices combine into a dead end. | At merge, run `worktree-setup` twice with the same branch against a dirty baseline; the second run must emit an actionable remediation (the exact `git worktree remove` to run), not a bare non-zero. |
| 3 | product | The dirty-baseline halt is misattributed as `/build` breakage: if `worktree-setup`'s failure output doesn't frame the failure as *pre-existing* baseline breakage, the persona blames `/build` — re-introducing, at the human layer, the exact misattribution the fresh-context loop exists to remove. | At merge, read the dirty-baseline failure message; it must state the baseline was already failing before any task ran, and name the failing baseline command. |
| 4 | technical | `verify` and `build-skill`'s foreman disagree on the not-yet-fixed Done-means input shape (Open question 2: JSON vs. line-oriented). `verify` then mis-parses — skips unrecognized items or mis-tiers a `test-backed` item as `script` — and reports PASS over an empty or partial item set. | At merge, run `verify` on an empty and on a malformed item list; it must fail closed (non-zero, "no items to verify"), never PASS vacuously. |
| 5 | technical | `verify`'s `probe`-tier check (exists + non-empty + optional pattern) is a weak proof with no freshness constraint of its own — a stale probe artifact from a prior attempt passes when no expected pattern is supplied, and the freshness gate lives only downstream in `evidence-capture`. | At merge, check whether `verify`'s probe branch applies any recency constraint to the supplied artifact, or relies solely on `evidence-capture` running afterward. |
| 6 | product | The status-flip duty (issue #14 lists it under the Scripts role; this story's acceptance criteria omits it) falls between stories: this doc defers it to `build-skill`'s design doc, and if `build-skill`'s doc assumes Scripts covered it, neither ships it and `/build` has no persisted task state. | At `build-skill`'s design review, confirm its doc explicitly claims or re-defers status-flip persistence; a silent doc means the gap realized. |
