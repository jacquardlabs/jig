| # | Lane | Failure mode | Detection hint |
|---|------|--------------|----------------|
| 1 | technical | Sign-off false positive: Step 1 treats a `## Revision History` heading as "design signed off," but viva appends that heading on *every* completed review round — including rounds that ended REVISED (changes requested), not only a DESIGNED sign-off. A doc mid-revision carries the heading yet is not approved, so the coach recommends `/plan` prematurely. | Spot-check the built `skills/coach/SKILL.md` sign-off detection against a design doc left in REVISED state that carries a `## Revision History` heading — confirm it distinguishes "heading present" from "last verdict was DESIGNED/approved," not just heading presence. |
| 2 | technical | Grammar drift from the real scripts: Step 1 reads `status-flip`'s `SUFFIX_RE` (` [PASS]`/` [REPLAN]`/` [ESCALATE]`) and the `status-flip: task <N> -> REPLAN|ESCALATE` commit subject with its `--reason` body. If the built coach's parse patterns are written from the doc's paraphrase rather than the landed `scripts/status-flip`, a suffix or commit-format change silently misreads task state. | Diff the coach's suffix/commit-parse patterns against `scripts/status-flip`'s actual `SUFFIX_RE` (line 55) and `commit_message` template (line 183); confirm the reason is read from the commit body, not `PLAN.md`. |
| 3 | technical | gate-ledger CLI surface assumed, not pinned: Step 1 calls `gate-ledger gate-get --branch <branch>` and `gate-ledger status`. If the installed studious exposes different subcommand names or JSON shapes, the recorded-verdict read fails or misreports and gate-touching recommendations degrade wrongly. | Confirm the built coach's `gate-ledger` invocations match the studious version actually installed, and that a `command -v gate-ledger` miss is exercised in a demo (states "studious not installed," never assumes a gate passed). |
| 4 | process | Pocock rule eroded under a "just do it all" user: the design forbids chaining (one confirmation, one dispatch) and requires re-assessment from fresh repo evidence, not memory. A build could drift into queuing the next dispatch off the same confirmation or auto-consuming a dispatched skill's verdict into a second dispatch — the auto-chaining PRODUCT.md rules out. | Confirm no path in the built `SKILL.md` dispatches a second skill without a fresh human confirmation, and that post-dispatch reassessment re-runs Step 1 (re-reads the repo) rather than reusing prior state. |
| 5 | product | Evidence-first safety bypassed in practice: the design's only guard against a misread is the human reading the printed evidence lines *before* confirming. If the built output buries evidence below the recommendation, or the confirm question precedes the evidence, a state misread dispatches the wrong skill unchecked. | Confirm the built output prints state + per-claim evidence lines before the recommendation and the confirmation question; confirm the repo-contradicts-conversation demo surfaces the conflict by name rather than silently following one side. |
| 6 | product | Silent gate skip on a standalone install: the degradation rule requires every gate row to name what is skipped and why. The risk is a built path that, without studious, jumps from all-`[PASS]` straight to `/finish` without stating "gate-audit / gate-acceptance skipped — studious not installed," leaving the human believing the work was gate-reviewed. | Walk the no-studious all-PASS state in a demo; confirm the recommendation names each skipped gate and its reason explicitly, never omits it silently (Standalone-capable principle). |
| 7 | technical | Ambiguity resolved by guess instead of asked: Step 1's third hard rule says two designs / more than one plan-shaped file / an unclear slug must stop and ask once, by name. A built coach could still pick a "most recent" candidate and proceed, converting an ambiguous state into a confident wrong recommendation. | Confirm the two-designs-without-plans case in a demo stops and asks by name; confirm no routing row silently resolves multiplicity by recency or first-match. |
