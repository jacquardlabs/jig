| # | Lane | Failure mode | Detection hint |

| 1 | technical | The method-existence check's "created by an earlier task" clause matches a method path as a plain substring of an earlier task's `Do:` line, so a path named-but-never-created passes because its text happens to appear as narrative in an earlier `Do:` (e.g. "unlike `scripts/verify`, this task…"). A genuinely missing method clears `method-not-found` on a coincidental substring — the design's own Open Question #3. | Inspect how the built `scripts/plan-lint` matches the "earlier task" clause (bare `in`/substring vs. an anchored backtick-span or whole-token match); confirm the broken fixture exercises a method that is *referenced* in an earlier `Do:` line but not *created* there, and that `method-not-found` still fires. |
| 2 | technical | The in-scope worked example added to `DESIGN.md`'s Formatting section and `skills/build/SKILL.md`'s Done-means template line lands with a tier-annotation string the `plan-lint` regex does not actually accept — a spacing, backtick-placement, or immediately-after-the-tier-word difference between the documented shape and the enforced shape. The epic pre-mortem's risk #1 (schema divergence) recurs inside this single story, at the doc-vs-code seam. | Diff the exact `(tier: script `path`)` string in DESIGN.md's added example and SKILL.md's template line against the tier regex in `scripts/plan-lint`; paste each documented example through the linter and confirm it lints clean. |
| 3 | technical | The "prose-only" edit to `skills/build/SKILL.md`'s Done-means template line shifts how `/build`'s Step 2.5 Foreman transcribes each item's `command` — SKILL.md is a model-consumed prompt, so a changed template line can change `/build`'s runtime behavior even with no code change, violating the epic goal's "consumes without modification." | At audit, compare the edited Done-means line against `/build`'s Step 2.5 transcription instructions; confirm the alternation semantics the Foreman relies on survive, and run `/build`'s existing scaffold/format tests against the edited SKILL.md. |
| 4 | technical | The task-splitting heading regex (with its tolerated optional `[PASS]`/`[REPLAN]`/`[ESCALATE]` suffix) mis-parses a heading — an em-dash title containing bracketed text, or a heading whose suffix is malformed — and silently drops that task's `Done means` checks, so a broken task passes because it was never linted at all. | Add a fixture task whose title itself contains bracketed or `###`-adjacent text plus a status suffix; confirm all of that task's `Done means` items are still counted and checked, not skipped. |
| 5 | product | A quick-path hand-author (PRODUCT.md CUJ 2, and this design's own journey step 3) trips `read-first-unresolved` / `method-not-found` / `load-bearing-cap-vague` without grasping the new backtick convention — the only extant hand-authored plan (the M0 dogfood) predates it and would fail wholesale, and category-named violation lines don't themselves teach "add backticks." The author abandons the linter instead of adopting the format. | At acceptance, run `plan-lint` against a naive prose-only single-task plan and read the printed violation lines as the author would: do they say what to change (backtick the path), or only name the category? Confirm DESIGN.md's Formatting example is discoverable from the error, not only from reading the design doc that dies at merge. |
| 6 | product | The deliberately-broken fixture cannot surface all eight categories as *distinct* findings in one run because violations mask one another — a zero-item task cannot simultaneously demonstrate `cap-count`, `hold-count`, and `item-count` as three separate findings, and a mis-parsed heading (see #4) drops a whole task's checks — so the "one distinct failure per violation category" acceptance demonstration silently under-delivers. | Count the distinct category names in the broken fixture's actual single-run output; require all eight (`cap-count`, `hold-count`, `item-count`, `invalid-tier`, `method-not-found`, `read-first-unresolved`, `not-here-followup-undrafted`, `load-bearing-cap-vague`) to appear, each on its own task, with none suppressed by an earlier failure on the same task. |
