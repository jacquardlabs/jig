| 1 | technical | The explicit `--split-on '(?i)^(Task \d+\|Not-here follow-ups\|Revision History)'` pattern the design commits viva to drifts from the heading text Step 4 actually emits. Step 4 reuses `skills/build/SKILL.md`'s grammar; if a task heading is worded anything other than `Task N…` (a localized title, a `Step N` variant, or a numberless heading), the anchored `^Task \d+` alternative silently fails to match, viva re-merges the cards, and issue #23's absorption bug reappears against `/plan`'s *own* output — the exact regression this story exists to close. | At merge, diff the literal `--split-on` string baked into `skills/plan/SKILL.md` against the `### ` heading text Step 4 produces; run the demonstration `PLAN.md` (acceptance demo 1) through viva and assert card count == task count + preamble + Not-here-follow-ups, with the follow-ups card byte-intact. |
| 2 | technical | Step 5's "silent internal lint-revise loop" is specified as "revise its own draft and re-lint" with no stated bound or oscillation guard. A model revision that fixes violation A while reintroducing B (or that "fixes" a `load-bearing-cap-vague` / `read-first-unresolved` finding by deleting the offending item rather than backticking it) either churns unboundedly or silently ships a plan stripped of legitimate content — invisible to the human by the design's own "loop is internal, invisible" posture. | At audit, check the built skill has a bounded retry / no-progress escape on the lint loop (not an open "revise until exit 0"); confirm the demonstration `PLAN.md` was not silently reduced in cap/hold items between first draft and final lint-clean state. |
| 3 | product | `DESIGN GAP` carries three distinct sub-causes (falsified assumption / missing test-or-lint infra / missing probe infra), each with a *different* resume action (revise the doc vs. install a tool vs. fix an inventory claim). If the built skill emits a generic or bare `DESIGN GAP`, the developer cannot tell which of three actions to take — collapsing the "recommend one concrete action" the verdict promises into a dead-end. Acceptance demo 4 exercises only the probe-infra cause. | At acceptance, trigger each of the three causes and confirm three distinct messages, each naming the specific cause and its own resume action; verify the assumption-falsified and missing-test-runner causes are demonstrated, not only the probe case demo 4 covers. |
| 4 | technical | The Input step reads any design doc "by content, not heading grammar." A hand-authored doc that omits or buries a load-bearing constraint (states a behavioral assumption only inside an "Alternatives" aside, or has no explicit constraints section) is silently misread, and `/plan` drafts a full spine against a wrong understanding instead of reporting `DESIGN GAP` — the failure the semantic-read flexibility trades away. | At merge, check `/plan` surfaces which claims/sections it extracted (or asks once when a required concept — problem, approach, risks — is absent) rather than defaulting silently; test with a doc that omits an explicit constraints/assumptions section and confirm it does not fabricate one. |
| 5 | product | Step 1b's probe-tooling inventory false-negatives. Signals (a) CLAUDE.md, (b) dependency manifest, (c) known config file are the only detection paths; a target with Playwright installed globally, declared in a monorepo-root manifest above the planned package, or under a non-standard config name is reported "no scripted-probe tool," forcing a spurious `DESIGN GAP` that blocks a legitimate probe-tier plan with no override. | At merge, confirm signal (a) — the CLAUDE.md read — is authoritative and lets a human clear a false `DESIGN GAP` by naming the tool in CLAUDE.md (a documented escape hatch), so the missing-probe verdict is reversible, not a hard dead-end when tooling exists but is undeclared where (a)–(c) look. |
| 6 | technical | The design leans entirely on each task's `Rests on:` line as the spine's load-bearing record (no separate spine artifact, by deliberate choice). If Step 4 writes a `Rests on:` value that references a task by a label `scripts/plan-lint`'s LOAD-BEARING derivation (or `/build`'s failure routine) does not key on, the mechanical dependency graph is silently wrong — a downstream consumer mis-derives which tasks are load-bearing with no error. | At merge, run the demonstration `PLAN.md` through `plan-lint` and confirm its LOAD-BEARING derivation matches `/plan`'s intended spine; check `Rests on:` values reference tasks by the exact identity token `plan-lint` and `/build` parse, not a free-text description. |
| 7 | product | The hard viva dependency (Step 6, no degraded path by design) produces an opaque failure when viva is not installed.
