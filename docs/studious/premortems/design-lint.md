| # | Lane | Failure mode | Detection hint |
|---|------|--------------|----------------|
| 1 | product | Check 5 enforces the `(qN)` fork-citation shape, which is one paper-dogfood author's shorthand, not a convention the sibling `design-skill` story commits to. If `design-skill` emits forks in any other shape, Check 5 finds zero `(qN)` tags and vacuously passes — the acceptance criterion "every fork has a recorded ruling" goes unverified while the linter reports green (epic pre-mortem risk #1's "passes checks that don't enforce anything"). | Run `design-skill`'s own real `design-<slug>.md` output through the landed `design-lint` before `design-skill`'s audit; if the doc raises forks but Check 5 reports "0 forks found / clean", the convention diverged. |
| 2 | product | `design-lint` lands unwired — nothing invokes it, and the `/design`→`design-lint` trigger is explicitly deferred to `design-skill`. If `design-skill`'s design never wires it in, the linter is dead code that never runs against a real doc and malformed `design-<slug>.md` still reaches viva. | At `design-skill`'s design-review/acceptance, confirm its doc names `design-lint` as an explicit step (auto-invoked or human/CI-run gate), not silence. |
| 3 | product | Check 4's failure-path detection is a fixed token vocabulary ("error", "fails", "refuses", …). An `Experience` section that genuinely covers a failure path but phrases it without a listed token passes as a false negative; conversely a drafter learns to keyword-stuff a token rather than actually narrate a failure path. | At build, confirm a test fixture covers a real failure path phrased WITHOUT any canned token and that the accept/reject behavior is deliberate — or that the vocabulary is documented as intentionally narrow. |
| 4 | technical | Check 3 resolves Assumptions-cited paths against the real checkout at lint time. If `design-lint` runs from a working directory other than the repo root (a `/design` scratch dir, a CI checkout whose root differs), a valid path citation resolves as missing and the check fails spuriously; a `../`/absolute citation could also escape or crash the resolver. | Inspect the path-resolution code for the base directory it resolves against (repo root vs. cwd vs. doc dir) and confirm the tests pin it; a `../` or absolute-path citation should be handled, not crash. |
| 5 | technical | Check 1's Revision-History exclusion assumes a trailing `---` + `## Revision History`. The design doc cites this as a two-example convention, but only the dogfood fixture actually exhibits it (`docs/design/finish-skill.md` ends with `## Open questions`, no trailer). A real doc that omits the trailer, or uses `---` as an in-body horizontal rule, mis-slices the section splitter and fires the 5-8 count check wrong. | Confirm the parser splits on the LAST `--- / ## Revision History` pair specifically, not any `---`; add a fixture with an in-body `---` inside a section. |
| 6 | technical | Each check hardcodes a build-phase constant this doc leaves unspecified (min inline-code-span count, ruling-verb list, failure-token list). Too loose and checks pass on prose-only/unruled docs (enforcing nothing); too strict and they fail the one real conformant fixture. | Confirm the clean-pass test runs against the actual dogfood fixture (or a faithful copy) and each violation fixture toggles exactly one check — a threshold that fails the real fixture is caught by the clean-pass test. |
| 7 | technical | Check 2 accepts inline-code spans as the concreteness marker. A `Contracts` section of backtick-quoted filler words (`` `maybe` ``, `` `later` ``) that name no real artifact passes the shape-marker heuristic while remaining prose-only in substance. | Inspect whether the count requires artifact-shaped spans (path/endpoint/field) or any backtick; a fixture of backtick-quoted filler that passes Check 2 exposes the gap. |
